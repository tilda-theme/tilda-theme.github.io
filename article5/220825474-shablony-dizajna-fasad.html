Шаблоны дизайна: фасад...

Code OOP
Tutorial

Когда дело доходит до шаблонов проектирования, у вас могут возникнуть вопросы:

Почему мы должны использовать шаблоны проектирования в программировании? Наш код итак может отлично работать.

На это мой встречный вопрос будет: «Вы предпочли бы жить в роскошном доме или в простом заведении с четырьмя стенами?» В конце концов, оба служат нашей цели.

Как правило, мы ищем роскошный дом, потому что он предлагает лучшие удобства и требует меньшего обслуживания, и само обслуживание может быть выполнено с меньшими затратами, поскольку базовая основа уже существует.

То же самое относится и к программированию: код, который использует шаблоны проектирования, легко понять, прост в обслуживании и прост в распространении.

В этой серии руководств мы рассмотрим некоторые различные шаблоны проектирования, которые доступны нам при программировании. Вы узнаете об их плюсах и минусах, а также о факторах, указывающих, где мы должны их использовать.

Во всех этих руководствах я буду использовать PHP в качестве базового языка для демонстрации шаблонов проектирования; Однако они на самом деле представляют собой концепцию, которая может применяться к любому языку программирования - это просто вопрос изменения синтаксиса в соответствии с вашим предпочтительным языком.

Правила проектирования разделены на четыре категории:

  • порождающие шаблоны
  • структурные шаблоны
  • поведенческие шаблоны
  • параллельные шаблоны

В этом уроке мы рассмотрим шаблон проектирования фасад. Он подпадает под категорию структурных шаблонов, поскольку он касается того, как ваш код должен быть структурирован, чтобы сделать его легко понятным и поддерживать его в хорошем состоянии в долгосрочной перспективе.

Шаблон проектирования фасад

UML

Проблема

Предположим, что вы выполняете несколько операций в последовательности и что одно и то же действие требуется в нескольких местах вашего приложения. Вы должны поместить тот же код снова и снова в разных местах. Вы это сделали, но через несколько дней вы обнаружите, что в этом коде нужно что-то изменить.

Вы видите проблему? Мы должны представить изменения во всех местах, где присутствует этот код. Это будет весьма болезненно, не так ли?

Решение

В качестве решения вы должны создать ведущий контроллер, который обрабатывает весь повторяющийся код. С вызывающей точки зрения мы просто вызываем ведущий контроллер для выполнения действий, основанных на предоставленных параметрах.

Теперь, если нам нужно внести какие-либо изменения в процесс, нам просто нужно будет изменить ведущий контроллер вместо того, чтобы производить изменения во всех местах, где мы использовали этот код.

Пример

В этом руководстве давайте выберем один урок, чтобы он стал более читаемым. Предположим, что вам поручено запланировать брак вашего друга. Если вы делаете все сами по себе, тогда представьте, что вам нужно покрыть. Это создаст более высокую вероятность ошибки и увеличит шанс пропустить что-то, что может кардинально повлиять на свадьбу вашего друга.

В этом случае вместо того, чтобы делать все самостоятельно, вы должны использовать планировщика свадьбы и следить за тем, чтобы работа выполнялась хорошо управляемым образом с меньшим шансом на ошибку.

Здесь вы ведете себя как клиент, который инициирует этот процесс, а свадебный планировщик работает как «фасад» для вас, выполняя задание, основанное на вашем направлении.

Пример кода

В этом разделе мы увидим еще один пример, который очень распространен для веб-сайтов, конечно, с примером кода. Мы увидим реализацию шаблона проектирования фасада на процессе проверки продукта. Но прежде чем проверять совершенный код с помощью шаблона фасада, давайте посмотрим на некоторый код, который имеет проблему.

Простой процесс проверки включает в себя следующие шаги:

  1. Добавить продукт в корзину.
  2. Рассчитать стоимость доставки.
  3. Рассчитать скидку.
  4. Создать заказ.

Проблема

// Simple CheckOut Process$productID = $_GET["productId"];$qtyCheck = new productQty();if($qtyCheck->checkQty($productID) > 0) {      // Add Product to Cart$addToCart = new addToCart($productID);// Calculate Shipping Charge$shipping = new shippingCharge();$shipping->updateCharge();// Calculate Discount Based on $discount = new discount();$discount->applyDiscount();$order = new order();$order->generateOrder();}

В приведенном выше коде вы обнаружите, что процедура покупки включает в себя различные объекты, которые необходимо создать для завершения операции покупки. Представьте, что вы должны реализовать этот процесс в нескольких местах. Если это так, это будет проблематично, когда код необходимо изменить. Лучше внести эти изменения сразу во все места.

Решение

Мы напишем то же самое только с шаблоном фасада, который делает тот же код более удобным и расширяемым.

class productOrderFacade {    public $productID = "";public function __construct($pID) {$this->productID = $pID;}public function generateOrder() {if($this->qtyCheck()) {// Add Product to Cart$this->addToCart();// Calculate Shipping Charge$this->calulateShipping();// Calculate Discount if any$this->applyDiscount();// Place and confirm Order$this->placeOrder();}}private function addToCart () {/* .. add product to cart ..  */}private function qtyCheck() {$qty = "get product quantity from database";if($qty > 0) {return true;} else {return true;}}private function calulateShipping() {$shipping = new shippingCharge();$shipping->calculateCharge();}private function applyDiscount() {$discount = new discount();$discount->applyDiscount();}private function placeOrder() {$order = new order();$order->generateOrder();}}

На данный момент у нас есть готовый фасад заказа продукта. Все, что нам нужно сделать, это использовать его с несколькими каналами связи кода, а не с кучей кода, как указано в предыдущей части.

Пожалуйста, проверьте количество кода ниже, который вам нужно будет использовать чтобы можно было использовать процесс покупки в разных местах приложения.

// Note: We should not use direct get values for Database queries to prevent SQL injection$productID = $_GET["productId"];// Just 2 lines of code in all places, instead of a lengthy process everywhere$order = new productOrderFacade($productID);$order->generateOrder();

Теперь представьте что будет, когда вам нужно внести изменения в процесс оформления заказа. Теперь вы просто создаете изменения в фасадном классе, который мы создали, вместо того, чтобы вносить изменения в каждое место, где до этого был этот код.

Вывод

Проще говоря, можно сказать, что шаблон фасад должен применяться в ситуации, когда вам нужен один интерфейс для выполнения нескольких процедур, как на примере планировщика свадьбы, который работает для вас как фасад, чтобы выполнить несколько процедур.
