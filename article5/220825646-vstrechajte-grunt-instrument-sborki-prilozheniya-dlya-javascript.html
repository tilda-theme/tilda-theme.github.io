Встречайте Grunt: инструмент сборки приложения для JavaScript ...

Code Web Development
Tutorial

Если вы работаете над большим проектом, у вас, без сомнения, есть сценарий сборки или множество сценариев задач, чтобы помочь вам с некоторыми повторяющимися частями рабочего процесса. Вы можете использовать Ant или Rake, в зависимости от языка, на котором написан проект.

Но что вы будете используеть, если проект в основном написан на JavaScript? Вот проблема, которую Бен Алман решил решить, когда создал Grunt.

Что такое Grunt?

Что такое Grunt? В README на Github написано что

Grunt - это инструмент для сборки командной строки на основе задач для проектов JavaScript.

Вот идея: при работе над проектом JavaScript есть множество задач, которые вы захотите делать постоянно. Какие, спросите вы? Ну, например такие как объединение данных файлов, запуск JSHint по вашему коду, запуск тестов или минимизация ваших скриптов. Если вы вставляете свой JavaScript в JSHint онлайн, вы, вероятно, понимаете, что есть лучший способ сделать это; даже если вы используете cat для конкатенации файлов или минификатором из командной строки, было бы неплохо иметь общий, единый набор команд для всех этих дополнительных задач, которые работали бы для каждого проекта JavaScript, не так ли?

Вот к чему стремится Grunt. Он имеет множество встроенных задач, которые так предоставляют возможностью создавать собственные плагины и скрипты, расширяющие основные функции.

Для получения дополнительной информации о введение в Grunt см. статью Бена в своем его блоге и блоге Bocoup.

Как установить Grunt?

Grunt написан на Node.js и доступен в виде пакета через менеджер пакетов Node (npm). Скорее всего вы захотите установить его глобально, поэтому используйте эту команду:

npm install -g grunt

Вы увидете, что он устанавливает довольно много зависимостей; есть другие пакеты npm, которые использует Grunt. Как только все будет сделано, мы готовы приступить!

Как использовать Grunt?

Как вы знаете, Grunt - это инструмент командной строки; поэтому для остальной части этого учебника, я предполагаю, что у вас есть открытое окно терминала.

Начнем с создания образца проекта каталога; мы не собираемся собирать здесь весь проект, но мы сможем увидеть, как работает Grunt в этом каталоге. Как только вы попадете в этот каталог, запустите команду grunt (согласно документации, если вы находитесь в Windows, вам может потребоваться запустить grunt.cmd). Вероятно, вы увидите что-то вроде этого:

<FATAL> Unable to find "grunt.js" config file. Do you need any --help? </FATAL>

Прежде чем вы сможете максимально эффективно использовать Grunt, вам понадобится файл grunt.js в каталоге проекта. К счастью, Grunt может автоматически генерировать файл grunt.js и другой материал скелета проекта - с помощью команжы задачей init, которая может работать без файла grunt.js. Но grunt init по-прежнему недостаточно, чтобы запустить ваш проект. Вам нужно выбрать тип проекта для генерации. Запуск grunt init предоставит вам список типов проектов на выбор:

  • jquery: плагин jQuery
  • node: модуль Node
  • commonjs: модуль CommonJS
  • gruntplugin: плагин Grunt
  • gruntfile: файл Grunt (grunt.js)

Если ваш проект действительно не соответствует ни одному из первых четырех типов проектов, вы можете использовать последний: gruntfile: он просто создает базовый файл grunt.js, который вы можете заполнить. Итак, давайте попробуем Grunt в действии на примере шаблона плагина jQuery. Запустите grunt init: jquery в вашем терминале.

Вы увидите много начального выхода. Если вы потратите время на чтение заметок шаблона, вы увидите, что нам нужно будет заполнить несколько значений, таких как название проекта и заголовок проекта. Фактически, после этой заметки вы увидите что-то вроде этого:

Please answer the following:[?] Project name (jquery.demo)

Всякий раз, когда вы инициализируете проект, Grunt задает вам ряд вопросов, поэтому он может заполнить несколько вариантов. Что такое значение в круглых скобках? Это предложение по умолчанию, основанное на типе проекта и имени каталога проекта. Если вы хотите изменить его, напишите собственное имя проекта в конце строки и нажмите «enter»; в противном случае просто нажмите «enter», чтобы использовать имя по умолчанию.

Продолжайте и заполните остальные поля. Вот что еще вам нужно для его создания проекта плагина jQuery:

  • Название проекта
  • Описание
  • Версия
  • Git репозиторий проекта
  • Главная страница проекта
  • Трекер проекта
  • Лицензии
  • Имя автора
  • Электронная почта автора
  • URL-адрес автора
  • Требуемая версия jQuery

Многие из них имеют значения по умолчанию; если вы хотите использовать значение по умолчанию, просто нажмите enter для этой строки; чтобы оставить поле пустым, вы можете просто набрать «none». После того, как вы просмотрите все параметры, вы увидите, что Grunt создает некоторые базовые файлы проекта. Такие как эти:

LICENSE-GPLLICENSE-MITREADME.mdgrunt.jslibs|-- jquery|    |-- jquery.js|-- qunit     |-- qunit.css     |-- qunit.jspackage.jsonsrc|-- jquery.demo.jstest|-- jquery.demo.html|-- jquery.demo_test.js

Как вы можете видеть, это дает нам хорошее начало: у нас есть не только наш файл плагина (src/jquery.demo.js), но и тесты Qunit (test/jquery.demo_test.js). И это тоже не пустые файлы. У них есть начальный контент, с таким супер-базовым плагином jQuery и модульными тестами. Идите и проверьте содержимое этих файлов, и вы увидите, что я имею в виду.

Grunt делает больше, чем настраивает проект для вас.

Конечно, Grunt делает больше, чем настраивает проект для вас. Примечательно, что в нашем проекте теперь есть grunt.js: файл конфигурации для конкретного проекта; из-за параметров, которые он устанавливает, мы теперь можем использовать другие встроенные задачи Grunt. Скоро мы раскроем его и внесем некоторые изменения, но теперь давайте запустим некоторые задачи.

Если вы запустите grunt без параметров, мы выполним задание по умолчанию, если оно было установлено. В случае проекта плагина jQuery это эквивалентно запуску этих четырех команд:

  • grunt lint: проверяет ваш JavaScript на JSHint
  • grunt qunit: запускает тесты Qunit
  • grunt concat: объединяет ваши файлы проекта и помещает новый файл в папку dist
  • grunt min: минимизирует вывод concat.

Здесь следует кое-что отметить о тестах Qunit: тесты Qunit предназначены для запуска в браузере по умолчанию; просто откройте файл tests/jquery.demo.html (или ваш эквивалент) в браузере. Тем не менее, grunt qunit захочет запустить их в терминале, а это значит, что вам нужно установить PhantomJS. Это не сложно: просто перейдите на phantomjs.org загрузите и установите последнюю версию. Если Grunt увидит что у вас установлен PhantomJs, он сможет запускать тесты Qunit из терминала.

Таким образом, после запуск grunt вы должны получить результат, похожий на это:

Как вы можете видеть, каждая из наших четырех задач запущена. Если какая-либо из них потерпит неудачу, остальные задачи будут отменены (если вы не назовете Grunt с флагом -force).

Как настроить мои задачи?

Сразу из коробки в Grunt доступен огромный функционал. Однако, давайте немного расширим его, открыв файл grunt.js и произведем некоторую настройку.

Внутри файла grunt.js вы увидите, что вся настройка выполняется путем передачи литерала объекта в grunt.initConfig(). Давайте рассмотрим несколько свойств нашего объекта конфигурации.

pkg

Это свойство указывает на файл package.json, который Grunt создал в нашем каталоге проекта. Наличие файла package.json является частью спецификации CommonJS Packages; это единственное место, где может хранится большинство метаданных о проекте (имя, версия, домашняя страница, ссылка репозитория ... многие из значений, которые вы устанавливаете при инициализации проекта). Однако это свойство pkg делает больше, чем просто указывает на файл пакета: обратите внимание на синтаксис: "<json: package.json>". Это одна из встроенных директив Grunt: она фактически загружает файл JSON, поэтому Grunt (или вы) может получить доступ ко всем свойствам в файле package.json из свойства pkg.

meta

Свойство meta - это объект с единственным свойством: баннер. Этот баннер - это комментарий, который находится в верхней части конкатенированных или минифицированных файлов проекта. Как вы можете видеть, это строка с некоторыми тегами шаблона (<% =%>); в большинстве случаев теги окружают вызов свойства в свойстве pkg, например pkg.title. Однако вы также можете выполнять функции внутри этих тегов: как например это демонстрирует grunt.template.today() и _.pluck().

concat/min/qunit/lint/watch

Я сгруппировал следующие пять свойств вместе, потому что они очень похожи. Все они задают параметры для конкретных задач, по которым они и были названы. При настройке этих задач важно отметить, что Grunt различает два типа задач: обычные задачи и многозадачность. В основном разница состоит в том, что обычные задачи имеют только один набор параметров конфигурации, тогда как многозадачность может иметь несколько наборов инструкций (называемых целями). Из пяти задач, которые я перечислял в заголовке этого раздела только watch не является многозадачным.

Обратите внимание, что в нашем конфигурационном объекте свойства qunit и lint являются объектами с свойством files. files - это единственная цель для этой задачи. В обоих случаях она представляют собой массив файлов, которые будут использоваться при выполнении этой задачи. Предположим, я хочу, чтобы я мог использовать только файлы из подкаталога src. Я мог бы добавить еще одну цель, чтобы свойство lint выглядело так:

lint: {   files: ["grunt.js", "src/**/*.js", "test/**/*.js"],  src: ["src/**/*.js"]},

Теперь, чтобы lint только файлы в src, я запускаю grunt lint: src: Я передаю имя цели после двоеточия. Если я запустил только grunt lint, то будут запущены обе цели.

В случае задач concat и min цели более сложные: это объекты с исходными (src) и целевыми (dest) свойствами. Конечно, эти свойства сообщают Grunt соответственно, где взять файлы и куда их поместить, когда они будут обработаны. Если вы добавите другие файлы в свой проект, вы захотите добавить их в нужное место, чтобы убедиться, что они объединены и правильно просчитаны. Итак, если бы я добавил файл src/utils.js, от которого зависел мой плагин jQuery, я бы сменил concat.dist.src на это:

src: ["<banner:meta.banner>", "src/utils.js", "<file_strip_banner:src/<%= pkg.name %>.js>"],

Рассматривая некоторые из этих задач более внимательно, вы заметите несколько других директив: наиболее важной является, вероятно, директива . Это позволяет вам получить доступ к свойствам других задач для повторного использования. Вы заметите, что конфигурация для задачи watch использует , так что она работает в том же списке файлов, которые мы предоставили для задачи lint. Вы можете узнать больше о других директивах в документации Grunt.

Говоря о watch задаче, что именно она делает? Очень просто: она запускает задачи в свойстве tasks при изменении любого файла в этом списке файлов. По умолчанию запускаются задачи lint и qunit.

jshint

Это свойство просто настраивает, какие «плохие части» JSHint ищет в вашем JavaScript. Полный список параметров можно найти на страницах параметров веб-сайта JSHint.

В самом низу нашего файла grunt.js вы увидите эту строку:

grunt.registerTask("default", "lint qunit concat min");

Это то, что создает нашу задачу по умолчанию; та, которая запускается, когда мы просто вызываем grunt. Это фактически создает алиас для задачи, и вы можете создать столько алисов, сколько хотите:

grunt.registerTask("src", "lint:src qunit:src concat:src min:src");

Предполагая, что вы создали цели src для каждой из этих задач, теперь вы можете вызвать grunt src и выполнить именно то, что вам нужно.

Как использовать сторонние задачи?

Хотя задачи, которые идут вместе с Grunt много всего могут сделать, но вы, вероятно, можете подумать и о других вещах, которые вы хотели бы автоматизировать. Не волнуйтесь: Grunt поставляется с API, который позволяет любому создавать задачи и плагины Grunt. Хотя мы не будем создавать какие-либо задачи Grunt в этом уроке, если вы заинтересованы в этом, вы должны начать с шаблона плагина Grunt (выполните команду grunt init: gruntplugin), а затем прочитайте документы API. После того, как вы написали свою задачу, вы можете загрузить ее в проект, добавив эту строку в файл grunt.js вашего проекта:

grunt.loadTasks(PATH_TO_TASKS_FOLDER);

Обратите внимание, что этот параметр не является путем к самому файлу задачи, это путь к папке, в которой находится файл задачи.

Однако другие плагины Grunt начинают появляться, а некоторые доступны на NPM. После установки через npm install вы можете загрузить их в свой проект с помощью этой строки:

grunt.loadNpmTasks(PLUGIN_NAME);

Конечно, вы должны проверить документацию плагина, чтобы узнать, что вы должны добавить в свой объект конфигурации.

Какие плагины Grunt доступны? Ну, так как Grunt настолько новый (меньше, чем месяц, когда я пишу это), их еще не так много. Я нашел два:

  • grunt-css: для литирования и минимизации CSS
  • grunt-jasmine-task: для запуска спецификаций Jasmine

Если вы нашли другие, я хотел бы услышать о них; опубликуйте их в комментариях!

Заключение

Хотя Grunt - очень новый проект, он уже достаточно самодостаточный; как мы видели, он поставляется с почти всем, что вам может быть нужно, чтобы использовать его в большом проекте и более того может быть даже расширен.

Я надеюсь, что Grunt станет стандартом сообщества, и мы увидим в ближайшем будущем множество задач, плагинов и шаблонов инициализации. А как вы к этому относитесь?
